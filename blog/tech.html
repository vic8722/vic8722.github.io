SQL Injection

April 3, 2016

SQL injection is a big problem for websites that allow users to make queries into database tables. It can allow a mischievous person to modify a search to gain access to information that is not intended for them to view, like user details, passwords, and account information. They can also potentially modify these databases, deleting data or modifying existing entries or adding new entries.

So how would one go about performing this devious action? First, there needs to be a place on a website that allows access to the SQL database. For example, a search bar that looks up product information and display results to the user. When a user types into the text box, a message is sent to perform a query on the SQL database, filling in the search text for a variable setup in the SQL query. Depending on how that was setup, it is possible to add in additional statements that will get evaluated outside the bounds of the parameter the search is intended to be evaluating.

There are a few methods that can be used to protect against SQL injection, namely blacklists, whitelists, and parameterized queries.

Blacklists

Blacklists involve providing a list of terms that can not be used in the search field, like "select," "insert," and "drop". However, as one would imagine, this can cause problems if one of the terms being prevented from use ends up being part of a legitimate search term. Additionally, there are ways to get around a search from showing up on a blacklist by utilizing a loop-hole where white-space characters can be represented by multiple hex values. There are also other methods for getting around the use of blacklists, which makes this the least desirable method for protection against SQL injection.

Whitelists

A white list is the opposite of a blacklist, instead of proving terms that can not be accepted and accepting all others, whitelists provide specific approved entries and prohibit all others. If you know that your desired input is 7 characters long and only contain numbers, you can specify those requirements. This would prevent anyone from being able to inject SQL in these type of fields. However, for our initial example of a general search box we will not be able to predetermine the structure of entries being searched for. This leads us to our final method for protection.

Parameterized Queries

Using a basic method for SQL queries has the entire statement being evaluated simply as part of the text. This is what allows a portion of the search that is expecting to evaluate one parameter to have malicious code inserted to add extra parameters and actions. If instead one sets up the SQL query to use a SqlParameter the term that you are getting input for can be separated out into a specific parameter. Then when the search is sent to the SQL Server it is sent as a contained parameter instead of being part of the main text to be evaluated.

Conclusion
